# Lab 6: AC97-lite PCM Audio

Generated folder: `labs/audio/`

Lab 6 introduces audio as a stream of samples. Instead of drawing pixels into a
framebuffer, you fill an audio buffer with PCM data and ask the runtime to play
it. The lab uses an AC97-like model, simplified enough for a course but close
enough to expose the important ideas: mapped buffers, sample rate, channels,
amplitude, playback length, and stop control.

## Why This Matters

Audio is unforgiving because timing is audible. A graphics program can drop a
frame and still look acceptable; an audio program that feeds the device at the
wrong rate produces pops, silence, distortion, or wrong pitch.

> [!NOTE]
> The music maker example builds on the same core idea as this lab: generate PCM
> samples from application state, then hand a buffer to the audio device.

## Device Model

| Concept | Meaning |
| --- | --- |
| PCM sample | numeric amplitude at one point in time |
| sample rate | number of samples per second |
| channel count | mono or stereo sample layout |
| buffer | mapped memory where samples are written |
| playback count | number of bytes the device should consume |

For the course path, a square wave is enough to prove that the buffer is mapped
correctly and that playback length matches the requested duration.

## Plan

1. Map the audio buffer.
2. Validate requested frequency and duration.
3. Convert milliseconds into sample count.
4. Generate a square wave into the buffer.
5. Start playback for the correct byte count.
6. Stop playback cleanly when requested.

> [!IMPORTANT]
> Do not write past the mapped audio buffer. Duration, sample rate, channel
> count, and bytes per sample must agree before you write.

## Requested Functions

- `int audio_map_buffer(void)`
- `int audio_fill_square_wave(uint32_t hz, uint32_t ms)`
- `int audio_play(size_t byte_count)`
- `int audio_stop(void)`

## Guided Gaps

- Decide how to reject zero frequency or zero duration.
- Compute samples per half-period without dividing by zero.
- Pick an amplitude that is audible without clipping.
- Keep byte count and sample count separate in your code.
- Decide whether `audio_play` should accept a count larger than the generated
  buffer.
- Make `audio_stop` safe to call even when nothing is playing.

> [!TIP]
> When the WAV sounds wrong, inspect the generated byte count first. Many audio
> bugs are unit bugs: samples, frames, channels, and bytes are not the same
> thing.

## Common Mistakes

- Using frequency where period is required.
- Treating milliseconds as samples.
- Writing one channel when the output expects stereo.
- Starting playback before filling the buffer.
- Forgetting that signed PCM values need both positive and negative amplitude.

## Discussion Prompts

- What changes if you generate sine waves instead of square waves?
- How would a music tracker represent notes, tempo, instruments, and playhead?
- Why do audio APIs often use ring buffers?
- What should happen if the application tries to play while another buffer is
  already active?
